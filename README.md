# Devtools

Проект демонстрирует настройку Gradle, работу с пакетами, record и юнит-тестирование.

---

## 🚀 Quick Start

### Запуск через Gradle Tool Window
1. Открой `View → Tool Windows → Gradle`
2. Разверни дерево:
   - `Tasks → application → run` — запуск приложения
   - `Tasks → verification → test` — запуск тестов
3. Дважды кликни на задачу или нажми иконку Run ▶️

### Запуск через Run Anything
- Нажми **Ctrl + Ctrl** (два раза)
- Введи:
  - `gradle run` — запуск приложения
  - `gradle test` — запуск тестов
  - `gradle build` — полная сборка проекта

### Запуск через терминал
```bash
./gradlew run
./gradlew test
./gradlew build
```

## Пакеты

**`ru.mentee.power`** — корневой пакет проекта.

**Зачем нужен пакет:**
- Предотвращает конфликты имён (классы с одинаковыми названиями из разных модулей не пересекаются)
- Организует код по функциональности
- Правило обратного доменного имени (`ru.mentee.power`) гарантирует глобальную уникальность

**Структура:**
```
src/
└── main/
    └── java/
        └── ru/
            └── mentee/
                └── power/
                    ├── ProgressDemo.java      # точка входа
                    └── MenteeProgress.java    # record с бизнес-логикой
src/
└── test/
    └── java/
        └── ru/
            └── mentee/
                └── power/
                    └── MenteeProgressTest.java   # JUnit-тесты
```

## Git локальный цикл

### Правило веток: feature/DVT‑X
- **master** — чистая ветка для стабильных версий (не коммитим напрямую)
- **feature/DVT‑X** — ветки под конкретные задачи (где X — номер урока)

### Локальный цикл (init → commit → branch)
```bash
# 1. Инициализация репозитория
git init

# 2. Проверка статуса
git status

# 3. Добавление файлов в staging
git add .

# 4. Коммит в master
git commit -m "Initial commit"

# 5. Создание feature-ветки
git checkout -b feature/DVT-3

# 6. Работа в ветке → коммиты
git add .
git commit -m "Добавить фичу"

# 7. Просмотр истории
git log --oneline
```


## Сценарий ручной проверки DVT-6

В рамках задачи DVT-6 была выполнена ручная проверка работоспособности функционала подсчета прогресса менти и валидации данных.

### 1. Сборка и запуск приложения
Запуск главного класса ProgressTracker и проверка консольного вывода суммарного прогресса:
$ ./gradlew run
* Ожидаемый результат: В консоль выводится строка с корректным расчетом: "Суммарно: пройдено 25 из 36 уроков, осталось 11 уроков".

### 2. Запуск автоматических тестов
Запуск разработанных Unit-тестов (включая тесты на валидацию record Mentee и граничные случаи подсчета прогресса):
$ ./gradlew test
* Ожидаемый результат: Тесты успешно пройдены, статус сборки: BUILD SUCCESSFUL.

### 3. Отладка и инспектирование переменных (Debug)
Для проверки корректности работы цикла в методе calculateTotalProgress была запущена отладка с точкой останова (breakpoint) внутри цикла while (на операции суммирования):
* Проверка вкладки Variables: На паузе внутри цикла проверено состояние счетчиков и индексов (idx, totalCompleted, totalTotal).

### 4. Динамическое вычисление выражений (Evaluate Expression)
На 3-й итерации цикла (idx == 2), когда данные по первым двум менти уже суммированы (totalCompleted == 13), выполнено выражение через Alt + F8:
totalCompleted + mentees[idx].completedLessons()
* Ожидаемый результат: Выражение возвращает точное значение 25 (13 + 12 за последнюю итерацию), подтверждая математическую корректность алгоритма до шага сохранения в переменную.

# 📐 Кодстайл-гайд (Checkstyle)

## 📏 Форматирование

| Правило | Требование |
|---------|------------|
| **Отступы** | 4 пробела (не табуляция!) |
| **Длина строки** | Максимум 100 символов |
| **Перенос строк** | Отступ для перенесённых строк — 8 пробелов |
| **Пробелы** | После `if`, `for`, `while`, `catch` — обязательно |
| **Фигурные скобки** | Открывающая на той же строке, закрывающая на новой |
| **Пустые блоки** | Только как `{}` (без пробелов внутри) |
| **Пробелы в конце строк** | Запрещены |

---

## 📦 Импорты

- **Без звёздочек** (`import java.util.*` — ❌)
- Сначала **статические** импорты, потом **обычные**
- Внутри группы — **по алфавиту**
- Пустая строка **между** группами импортов

---

## 🏷️ Именование

| Элемент | Формат | Пример |
|---------|--------|--------|
| **Классы, интерфейсы, enum** | `UpperCamelCase` | `UserService` |
| **Методы** | `lowerCamelCase` | `getUserById()` |
| **Переменные** | `lowerCamelCase` | `userName` |
| **Константы** | `UPPER_SNAKE_CASE` | `MAX_SIZE` |
| **Пакеты** | только строчные | `ru.mentee.power` |
| **Параметры** | `lowerCamelCase` | `userId` |
| **Типы-параметры (generic)** | одна заглавная буква | `T`, `E`, `K`, `V` |

**Важно:** Сокращения (ID, URL) — только **одна** заглавная буква:  
✅ `getUserId()` ❌ `getUserID()`

---

## 🧩 Структура класса

1. **Пакет** (`package`)
2. **Импорты**
3. **Пустая строка**
4. **Класс / интерфейс / enum**
5. **Поля** → **Конструкторы** → **Методы** (в этом порядке)

---

## 📝 Javadoc

- **Обязателен** для публичных (`public`) классов и методов
- Порядок тегов: `@param` → `@return` → `@throws` → `@deprecated`
- Первая строка — **краткое описание** (с заглавной буквы)
- Для удобства флагу severity было установлено значение "ignore"

---

## 🚫 Запрещено

| Что | Почему |
|-----|--------|
| `*` в импортах | Засоряет пространство имён |
| Голые `TODO:` | Только `TODO:` (двоеточие обязательно) |
| `switch` без `default` | Всегда должен быть |
| Пустые `catch` без комментария | Должна быть причина |
| `finalize()` | Устарел и опасен |

---

## 🧹 Остальное

- **Один оператор на строку**
- **Одна переменная на строку**
- **Массивы:** `int[] arr` (не `int arr[]`)
- **Модификаторы:** `public` → `protected` → `private` → `static` → `abstract` → `final`
- **Аннотации:** на отдельной строке (кроме `@Test` и однострочных)

# Code Review Checklist

## Функциональность
- [ ] Код решает поставленную задачу полностью
- [ ] Обработаны граничные случаи (null, пустые данные, экстремальные значения)
- [ ] Обработка ошибок реализована корректно

## Тесты
- [ ] Добавлены тесты для нового функционала
- [ ] Все тесты проходят локально: ./gradlew test
- [ ] Покрыты позитивные и негативные сценарии
- [ ] JaCoCo coverage >= 80% для нового кода

## Читаемость и стиль
- [ ] Имена переменных, методов и классов отражают назначение
- [ ] Нет дублирования кода (DRY)
- [ ] Checkstyle проходит без ошибок: ./gradlew checkstyleMain
- [ ] Нет закомментированного кода или отладочного вывода (System.out.println)

## Документация
- [ ] README обновлён (если добавлена новая функциональность)
- [ ] Публичные методы имеют JavaDoc (если применимо)
- [ ] Runbook обновлён (если изменились команды)

## Производительность и безопасность
- [ ] Нет очевидных проблем производительности
- [ ] Нет хардкода паролей, токенов или конфиденциальных данных

# Примеры Code Review комментариев

## ✅ Конструктивные комментарии

### Пример 1
**Проблема:** В методе `calculateTotal()` (строка 45) происходит валидация, расчёт и форматирование результата — три разные ответственности в одном методе.
**Почему это важно:** Нарушение SRP усложняет поддержку кода. При изменении логики расчёта легко случайно сломать валидацию или форматирование.
**Предложение:** Разделить на три метода: `validateInput()`, `performCalculation()` и `formatResult()`.

*Почему хороший:* Конкретное указание места, объяснение последствий и чёткое предложение решения.

### Пример 2
**Проблема:** В тесте `testUserCreation()` (строка 120) используется `Thread.sleep(1000)` для ожидания асинхронной операции.
**Почему это важно:** Фиксированные задержки делают тесты медленными и нестабильными (flaky tests). На CI может потребоваться больше времени.
**Предложение:** Использовать `Awaitility.await().atMost(5, SECONDS).until(...)` для явного ожидания условия.

*Почему хороший:* Указывает на конкретную проблему, объясняет почему это плохо в долгосрочной перспективе, предлагает проверенное решение.

## ❌ Токсичные комментарии

### Пример 3
**Проблема:** "Этот код написан ужасно, перепиши всё заново."

*Почему плохой:* Неконкретно, не указано что именно плохо. Токсичная формулировка "написан ужасно" вместо "в таком-то месте есть проблемы". Нет предложения по улучшению.

### Пример 4
**Проблема:** "Я бы никогда так не сделал, это неправильно."

*Почему плохой:* Выносит суждение о человеке, а не о коде. Не объясняет почему подход неправильный. "Неправильно" — субъективно без объяснения последствий. Нет альтернативного решения.

# Результаты само-ревью DVT-9

### Проблема 1: Отладочный вывод, не соответствующий бизнес-логике
**Файл:** `src/main/java/ru/mentee/power/devtools/ProgressDemo.java`
**Строка:** 19
```java
IO.println("Current branch: feature/DVT-3");
```

**Суть проблемы:** В коде присутствует отладочный вывод, который:
- Использует несуществующий класс `IO` (ошибка компиляции, если импорт не добавлен)
- Выводит информацию о ветке Git, которая не имеет отношения к бизнес-логике
- Является артефактом разработки, оставленным случайно

**Почему это важно:**
- Отладочный вывод в продакшн-коде засоряет логи и вводит в заблуждение
- Код не компилируется без дополнительного импорта `IO`
- Нарушает принцип чистоты кода — в решении должны оставаться только релевантные части

**Как исправить:** Удалить строку 19 целиком.

---

### Проблема 2: CI badge расположен перед заголовком README
**Файл:** `README.md`
**Строка:** 1

**Суть проблемы:** Бейдж статуса CI находится перед заголовком проекта, что нарушает стандартную структуру README.

**Почему это важно:**
- Нарушает визуальную иерархию — читатель должен сначала видеть название проекта, а не техническую информацию
- Во всех популярных open-source проектах принят порядок: заголовок → бейджи → описание
- Ухудшает восприятие документации с первого взгляда

**Как исправить:** Переместить бейдж под заголовок:
```markdown
# Devtools

[![Java CI](https://github.com/EGGGOOSE/devtools/actions/workflows/ci.yml/badge.svg)](https://github.com/EGGGOOSE/devtools/actions/workflows/ci.yml)
```

### Проблема 3: Неинформативное имя переменной в `ProgressTracker`
**Файл:** `src/main/java/ru/mentee/power/devtools/progress/ProgressTracker.java`
**Строка:** 10-14
```java
int index = 0;
int totalCompleted = 0;
int totalTotal = 0;

while (index < mentees.length) {
    totalCompleted += mentees[index].completedLessons();
    totalTotal += mentees[index].totalLessons();
    index++;
}
```

**Суть проблемы:** Имя переменной `totalTotal` не отражает суть — она хранит сумму всех уроков по всем mentee. Более того, имя `totalTotal` звучит как "итог итогов", что сбивает с толку. Также цикл while можно заменить на for-each для улучшения читаемости.

**Почему это важно:**
- Читаемость кода страдает из-за неинформативных имён
- Через полгода или для нового разработчика будет непонятно, что означает `totalTotal`
- Нарушение принципа "код должен быть самодокументируемым"

**Как исправить:**
```java
int totalCompleted = 0;
int totalLessons = 0;

for (Mentee mentee : mentees) {
    totalCompleted += mentee.completedLessons();
    totalLessons += mentee.totalLessons();
}
```

### Проблема 4: Отсутствие обработки null в `ProgressTracker.calculateTotalProgress`
**Файл:** `src/main/java/ru/mentee/power/devtools/progress/ProgressTracker.java`
**Строка:** 9

**Суть проблемы:** Метод не проверяет массив mentees на null. Если передать null, возникнет `NullPointerException` без понятного сообщения.

**Почему это важно:**
- Это публичный метод, который могут вызывать извне
- Отсутствие валидации входных параметров — частая причина багов
- Нарушает принцип "fail fast with clear message"

**Как исправить:** Добавить проверку в начало метода:
```java
public String calculateTotalProgress(Mentee[] mentees) {
    if (mentees == null) {
        return "Нет данных о прогрессе";
    }
    // ... остальной код
}
```
Или выбросить более информативное исключение:
```java
if (mentees == null) {
    throw new IllegalArgumentException("Массив mentees не может быть null");
}
```

### Проблема 5: Избыточный комментарий в `ProgressDemo`
**Файл:** `src/main/java/ru/mentee/power/devtools/ProgressDemo.java`
**Строки:** 10-12
```java
new MenteeProgress(
    name,
    sprint, // номер спринта
    hours   // запланированные часы на спринт
);
```

**Суть проблемы:** Комментарии избыточны — параметры `sprint` и `hours` уже имеют понятные названия в конструкторе `MenteeProgress(int sprintNumber, int plannedHoursPerWeek)`.

**Почему это важно:**
- Захламляют код визуальным шумом
- Хороший код должен быть самодокументируемым через осмысленные имена
- Избыточные комментарии могут устаревать и вводить в заблуждение

**Как исправить:** Удалить комментарии, оставив только:
```java
new MenteeProgress(name, sprint, hours);
```

### Проблема 6: Отсутствие тестов для `ProgressTracker` с пустым массивом
**Файл:** `src/test/java/ru/mentee/power/devtools/progress/ProgressLoopTest.java`

**Суть проблемы:** Нет теста для случая, когда в `calculateTotalProgress` передаётся пустой массив `Mentee[]`.

**Почему это важно:**
- Граничные случаи должны быть покрыты тестами
- Пустой массив — это валидный сценарий, который может встретиться в реальной работе
- Без теста можно случайно сломать обработку этого случая при рефакторинге

**Как исправить:** Добавить тест:
```java
@Test
@DisplayName("Пустой массив mentee — возвращает корректное сообщение")
void shouldHandleEmptyMenteeArray() {
    ProgressTracker tracker = new ProgressTracker();
    Mentee[] emptyArray = {};
    
    String result = tracker.calculateTotalProgress(emptyArray);
    
    assertThat(result)
            .contains("пройдено 0 из 0 уроков")
            .contains("осталось 0 уроков");
}
```
